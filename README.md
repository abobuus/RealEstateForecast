# Прогнозирование цен на недвижимость в России

Бакалаврский дипломный проект. Цель — построить модели машинного обучения, которые предсказывают цену квартиры на основе её характеристик и макроэкономических факторов.

---

## Данные

### Исходные датасеты

Два датасета с объявлениями о продаже квартир в России:

| Датасет | Период | Записей | Особенности |
|---------|--------|---------|-------------|
| `all_v2.csv` | 2018–2021 | 5.47M | Внутренние ID регионов платформы |
| `input_data.csv` | 2021 | 11.36M | Официальные коды регионов (КЛАДР) |

**Признаки:** дата, цена, площадь, площадь кухни, этаж, этажей в доме, количество комнат, координаты (широта/долгота), тип объекта, тип здания, регион.

### Вспомогательные данные

- `russia_regions.geojson` — границы 83 регионов РФ в формате полигонов (для определения региона по координатам)
- `russia_region_codes.csv` — справочник официальных кодов регионов РФ с названиями

## Что сделано

### Шаг 1 — Исследовательский анализ данных (EDA)

**Ноутбуки:** `01_eda_all_v2.ipynb`, `02_eda_input_data.ipynb`

Изучили каждый датасет отдельно: распределение цен, площадей, количество записей по регионам и годам, наличие пропусков и аномалий.

**Ключевые находки:**
- В `all_v2` регионы закодированы внутренними ID платформы (значения 3–61888) вместо официальных кодов — нужно исправить
- В `input_data` площадь кухни (`kitchen_area`) имеет отрицательные значения (~1.09M записей) — ошибки в данных
- Цены содержат выбросы: нулевые и свыше 1 млрд рублей
- `rooms = -1` встречается в обоих датасетах — это студии

---

### Шаг 2 — Исправление регионов

**Ноутбуки:** `03_region_geocoding_input_data.ipynb`, `05_region_geocoding_all_v2.ipynb`

**Проблема:** регионы в датасетах закодированы по-разному, и часть кодов неверная.

**Решение — Spatial Join:**
Для каждой записи берём координаты (широта/долгота) и определяем в каком регионе РФ она находится, используя полигоны границ регионов из GeoJSON. Затем заменяем старый код на официальный.

**Крым:** GeoJSON не включает Крым как регион России, поэтому ~7000 точек в Крыму не получили регион автоматически. Они определены вручную по географическому bbox-прямоугольнику:
- Республика Крым → код 91
- Севастополь → код 92

**Морские точки:** ~2454 точки из `all_v2` оказались в море рядом с побережьем Сочи — удалены как выбросы (квартира не может находиться в море).

**Результаты:**
- `input_data_clean.csv` — 11,358,150 записей, все регионы 1–92, подтверждено 100% совпадение с координатами
- `all_v2_region_fixed.csv` — 5,474,552 записей, все регионы 1–92

### Шаг 3 — Проверка качества (Sanity Check)

**Ноутбуки:** `04_sanity_check_input_data.ipynb`, `06_sanity_check_all_v2.ipynb`

После каждого преобразования данных проводится автоматическая проверка:
- Количество строк не изменилось сверх ожидаемого
- Нет NaN в ключевых колонках
- Регионы в допустимом диапазоне (1–92)
- Распределения цен и площадей не изменились
- Крым и Севастополь присутствуют

Все проверки пройдены успешно.

### Шаг 4 — Объединение датасетов

**Ноутбук:** `07_merge.ipynb`

Два датасета объединены в один. Перед объединением необходимо было решить проблему несовместимости признаков:

**`object_type` (тип объекта):**
Оба датасета содержат этот признак, но с разными кодами. Анализ статистики показал что это одно и то же:
- Вторичный рынок: код 1 (all_v2) = код 0 (input_data) → унифицировано в **0**
- Новостройки: код 11 (all_v2) = код 2 (input_data) → унифицировано в **1**

**`building_type` (тип здания):**
Кодировки полностью несовместимы — распределения значений слишком разные для надёжного сопоставления. Принято решение **удалить** этот признак, чтобы не вносить ошибки в данные.

**Итоговые колонки:** `date, price, level, levels, rooms, area, kitchen_area, geo_lat, geo_lon, object_type, region, source`

**Результат:** `merged.csv` — **16,832,702 записей**, период 2018–2021, 85 регионов.

## Эксперимент 1 — Базовые регрессионные модели ✅

**Ноутбук:** `10_experiment1_regression.ipynb`

**Разделение данных:**
- Train: 2018–2020, стратифицированная выборка **1.5M** записей (равномерно по годам)
- Test: 2021, случайная выборка **500k** записей
- Разделение по времени — чтобы модель не "видела будущее" при обучении

**Признаки:** area, kitchen_area, level, levels, rooms, object_type, region, dist_to_center, geo_lat, geo_lon, year, month

**Результаты:**

| Модель | R² | MAE | RMSE |
|--------|-----|-----|------|
| Random Forest | **0.7228** | 1.85M | 6.91M |
| LightGBM | 0.6958 | 1.97M | 7.24M |
| CatBoost | 0.6677 | 1.92M | 7.56M |
| XGBoost | 0.6559 | **1.81M** | 7.70M |
| Linear Regression | 0.1505 | 3.28M | 12.09M |
| Ridge | 0.1505 | 3.28M | 12.09M |
| Lasso | 0.1505 | 3.28M | 12.09M |

**Выводы:**
- Линейные модели плохо справляются (R²=0.15) — зависимость цены от признаков нелинейная
- Деревья и бустинг работают значительно лучше (R²=0.66–0.72)
- Random Forest показал лучший R²=0.72 на стратифицированной выборке 1.5M
- Средняя ошибка ~1.8M руб. (~20-30% от медианной цены) — есть куда расти
- Ожидается улучшение после добавления макроданных и инфраструктурных признаков

## Эксперимент 2 — Макроэкономические данные ✅

**Ноутбук:** `11_experiment2_regression_macro.ipynb`

К каждой записи добавлены макроэкономические показатели на дату сделки (джойн по месяцу и региону):

| Признак | Источник | Привязка |
|---------|----------|----------|
| `cbr_rate` | Ключевая ставка ЦБ РФ | по месяцу |
| `inflation` | Инфляция (ИПЦ) | по месяцу |
| `usd_rate` | Курс доллара (среднемесячный) | по месяцу |
| `income_mln` | Средний доход населения | по месяцу + региону |

**Результаты:**

| Модель | R² Эксп.1 | R² Эксп.2 | ΔR² | MAE Эксп.2 |
|--------|-----------|-----------|-----|------------|
| Random Forest | 0.7228 | 0.7210 | −0.0018 | 1.910M |
| LightGBM | 0.6971 | 0.6883 | −0.0088 | 1.887M |
| CatBoost | 0.6591 | 0.6669 | +0.0078 | 1.912M |
| XGBoost | 0.6559 | 0.6428 | −0.0131 | 1.880M |
| Linear Regression | 0.1505 | 0.2294 | **+0.0789** | 2.658M |

**Выводы:**
- Макроданные практически не улучшают качество деревьев (изменения в пределах ±0.01)
- Причина: признаки `year` и `month` уже являются прокси для макропоказателей. Каждому конкретному месяцу соответствует ровно одно значение ставки/курса/инфляции — дерево получает ту же информацию через сплит по времени. Добавление макропризнаков избыточно
- Единственный по-настоящему полезный макропризнак — `income_mln` (доходы по регионам, SHAP=0.96 млн руб.), так как он варьируется одновременно по времени и по регионам — информация, которой нет в `year`/`month`
- Линейные модели улучшились заметнее (+0.08 R²): им сложно самостоятельно извлечь временной тренд, поэтому явные числовые значения макропоказателей помогают
- Макроданные важны не для точности на обучающем периоде, а для **обобщаемости на будущие периоды**: при предсказании на 2022+ год модель не знает "что такое 2022", но зная ставку ЦБ и курс — сможет обобщить
